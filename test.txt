GIAI ĐOẠN 1: ĐẦU RA CỦA AI EXTRACTION (Raw LLM Output)
Cục này là do con AI (Gemini/GPT) khạc ra. Mày sẽ thấy nó chứa đầy "rác định dạng" (Hallucination) vì bản chất LLM xử lý ngôn ngữ tự nhiên.

JSON
{
  "ngay_bao_cao": "2026-03-02", 
  "tong_so_vu_chay": "Một vụ", 
  "tong_so_vu_cnch": 1,
  "danh_sach_cnch": [
    {
      "thoi_gian": "05 giờ 10 phút",
      "dia_diem": "Sông Sài Gòn",
      "luc_luong": "01 xe chỉ huy",
      "ket_qua_xu_ly": "Bàn giao thi thể"
    }
  ],
  "tong_co_so_kiem_tra": "0" 
}
Vấn đề của cục data này: tong_so_vu_chay đang là chữ "Một vụ" thay vì số nguyên. tong_co_so_kiem_tra đang là chuỗi "0" thay vì số 0. Ngày tháng đang ở format quốc tế YYYY-MM-DD. Nếu chốt cục này xuống DB là gãy cánh liền.

GIAI ĐOẠN 2: ĐẦU RA CỦA VALIDATION LAYER (Clean JSON)
Cục này đã đi qua bộ rây Pydantic (Agent Kiểm dịch). Chữ đã biến thành số, format đã được nắn lại. Đây là cục Data chuẩn mực được phép gọi lệnh INSERT để lưu vào cột JSONB của PostgreSQL.

JSON
{
  "ngay_bao_cao": "02/03/2026",
  "tong_so_vu_chay": 1,
  "tong_so_vu_cnch": 1,
  "danh_sach_cnch": [
    {
      "thoi_gian": "05 giờ 10 phút",
      "dia_diem": "Sông Sài Gòn",
      "luc_luong": "01 xe chỉ huy",
      "ket_qua_xu_ly": "Bàn giao thi thể"
    }
  ],
  "tong_co_so_kiem_tra": 0
}
Sự thay đổi: "Một vụ" đã ép thành số 1. "0" đã ép thành số 0. Ngày 2026-03-02 đã ép thành 02/03/2026.

GIAI ĐOẠN 3: ĐẦU RA CỦA AGGREGATION (Aggregated JSON Payload)
Bây giờ cán bộ trực ban muốn xuất Báo cáo Tuần. Thằng thư viện Pandas sẽ móc 7 cục Clean JSON của 7 ngày lên (Giai đoạn 2), cộng dồn (SUM) các cột số và nối (CONCAT) các danh sách lại với nhau.

Đây là cục Data "Bọc Thép" cuối cùng được nhả ra để đút vào file Word:

JSON
{
  "tu_ngay": "02/03/2026",
  "den_ngay": "08/03/2026",
  "tong_so_vu_chay": 3, 
  "tong_so_vu_cnch": 2,
  "danh_sach_cnch": [
    {
      "thoi_gian": "05 giờ 10 phút ngày 02/03",
      "dia_diem": "Sông Sài Gòn",
      "luc_luong": "01 xe chỉ huy",
      "ket_qua_xu_ly": "Bàn giao thi thể"
    },
    {
      "thoi_gian": "06 giờ 15 phút ngày 03/03",
      "dia_diem": "Cầu Phú Long",
      "luc_luong": "01 cano",
      "ket_qua_xu_ly": "Cứu được nạn nhân"
    }
  ],
  "tong_co_so_kiem_tra": 23
}
Sự thay đổi: Cục này là kết tinh của 7 ngày. tong_so_vu_chay nảy lên số 3. danh_sach_cnch bây giờ là một mảng bự chứa 2 vụ của ngày mùng 2 và mùng 3 ghép lại.

GIAI ĐOẠN 4: ĐẦU RA CỦA WORD EXPORT (Binary Stream)
Ở giai đoạn này, dữ liệu không còn là JSON nữa. Thư viện docxtpl lấy cục Aggregated JSON ở Bước 3, nhét vào các lỗ đục {{ }} trong file Word Template.

Đầu ra (Output): Nó đéo trả về text, nó trả về một luồng byte (Byte Stream) với Content-Type là application/vnd.openxmlformats-officedocument.wordprocessingml.document.

Kết quả cuối cùng: Trình duyệt của user sẽ nảy ra cái popup tải xuống file Bao_Cao_Tuan.docx. Khi mở bằng Microsoft Word, nó sẽ hiển thị thế này:

I. TÌNH HÌNH CHÁY, NỔ, CỨU NẠN VÀ CỨU HỘ

Tình hình cháy: Trong tuần xảy ra 3 vụ.

Tình hình cứu nạn, cứu hộ: Trong tuần xảy ra 2 vụ cứu nạn, cứu hộ, cụ thể:
3.1. Vào lúc 05 giờ 10 phút ngày 02/03 nhận tin báo...
3.2. Vào lúc 06 giờ 15 phút ngày 03/03 nhận tin báo...

1. Khi đọc file Word mẫu (Template Generation)
Khi người dùng tải lên file Word mẫu (ví dụ: Báo_cáo_tuần_mẫu.docx) có chứa các thẻ Jinja2 như {{ tong_so_vu_chay }} hoặc {% tr for row in bang_thong_ke %}, hệ thống không lưu file Word này vào Database.

Cơ chế xử lý: Module word_scanner.py sử dụng Regex quét tầng XML của file Word để tìm tất cả các thẻ biến số. Dựa trên tiền tố của biến (ví dụ tong_ suy ra là số, danh_sach_ suy ra là mảng), nó tự động nội suy ra một bộ JSON Schema và bộ Aggregation Rules (Luật cộng dồn).

Dữ liệu lưu vào Database (Bảng extraction_templates, cột JSONB):
Hệ thống sẽ lưu 2 cục JSON này để làm "thước đo" cho các bước sau:

JSON
// Cục 1: schema_definition (Dùng để ép AI bóc tách đúng định dạng)
{
  "fields": [
    {"name": "tong_so_vu_chay", "type": "number", "description": "Tổng số vụ cháy"},
    {"name": "danh_sach_cnch", "type": "array", "items": {
        "type": "object",
        "fields": [
          {"name": "thoi_gian", "type": "string"},
          {"name": "dia_diem", "type": "string"}
        ]
    }}
  ]
}

// Cục 2: aggregation_rules (Dùng để Pandas biết cách cộng dồn)
{
  "rules": [
    {"output_field": "tong_so_vu_chay", "source_field": "tong_so_vu_chay", "method": "SUM"},
    {"output_field": "danh_sach_cnch", "source_field": "danh_sach_cnch", "method": "CONCAT"}
  ]
}
2. Khi trích xuất từ PDF và Lưu Database (AI Extraction & Validation)
Khi ném một file PDF (Báo cáo ngày) vào hệ thống, quá trình này diễn ra qua 2 nhịp cực kỳ nghiêm ngặt để chống lỗi "ảo giác" (Hallucination) của LLM.

Nhịp 1 (AI Bóc tách): Backend lấy file PDF băm thành text/markdown, rồi ném đoạn text đó cùng với cục schema_definition ở trên vào Prompt của LLM (Gemini). LLM bị ép buộc chỉ được nhả ra JSON theo đúng cấu trúc.
Ví dụ AI nhả ra: {"tong_so_vu_chay": "Hai vụ", "danh_sach_cnch": [...]}

Nhịp 2 (Kiểm dịch & Lưu trữ):
Tuyệt đối không lưu raw JSON của LLM xuống DB. Dữ liệu đi qua class DataValidator (sử dụng thư viện Pydantic). Nó tự động nắn chữ "Hai vụ" thành số nguyên 2, chuẩn hóa ngày tháng.

Dữ liệu lưu vào Database (Bảng extraction_jobs, cột extracted_data kiểu JSONB):
Cục JSON siêu sạch này mới là thứ được lưu:

JSON
{
  "tong_so_vu_chay": 2,
  "danh_sach_cnch": [
    {
      "thoi_gian": "05 giờ 10 phút ngày 02/03/2026",
      "dia_diem": "Sông Sài Gòn"
    }
  ]
}
3. Đầu ra xuất file (Aggregation & Word Export)
Khi người dùng chọn 7 cái Báo cáo Ngày để xuất thành 1 Báo cáo Tuần.

Cơ chế xử lý (Pandas): Backend kéo 7 cục JSON từ bảng extraction_jobs lên RAM. Hàm pandas.json_normalize() đập phẳng dữ liệu. Dựa vào aggregation_rules (đã lưu ở Bước 1), hệ thống chạy lệnh SUM để cộng 7 cái tong_so_vu_chay lại với nhau, và chạy lệnh CONCAT để nối 7 cái mảng danh_sach_cnch thành một mảng dài.

Dữ liệu Payload cuối cùng (Aggregated Data):
Nó đẻ ra một cục JSON Tổng, cấu trúc mapping 1-1 với file Word:

JSON
{
  "tong_so_vu_chay": 14, 
  "danh_sach_cnch": [
    {"thoi_gian": "05:10 02/03", "dia_diem": "Sông Sài Gòn"},
    {"thoi_gian": "08:00 03/03", "dia_diem": "Cầu Phú Long"},
    // ... nối tiếp của các ngày khác
  ]
}
Xuất file Word (Headless Render):
Thư viện docxtpl trên server mở file Word Template gốc ra. Nó cầm cục JSON Tổng ở trên nhồi thẳng vào mã XML của Word. Chỗ nào là thẻ {{ tong_so_vu_chay }} sẽ bị ghi đè thành số 14. Chỗ nào là thẻ lặp {% tr for %}, nó sẽ nhân bản dòng trong bảng (Table Row) tương ứng với độ dài của mảng danh_sach_cnch.

Đây là GIAI ĐOẠN 1: NUỐT KHUÔN WORD VÀ SINH LÝ LỊCH (TEMPLATE INGESTION & SCHEMA GENERATION).

Giai đoạn này xảy ra khi cán bộ Tham mưu upload cái file Báo cáo mẫu (file trống, đã đục lỗ) lên hệ thống lần đầu tiên.

1. Dữ liệu Đầu vào (Vật lý)
Cán bộ ném vào hệ thống một file .docx (ví dụ: Bao_cao_tuan_KV30_mau.docx). Trên mặt giấy Word của file này có chứa các đoạn text xen lẫn mã Jinja2:

"Trong tuần xảy ra {{ tong_so_vu_chay }} vụ."
"Chi tiết sự cố: {%p for vu in danh_sach_cnch %} ... {%p endfor %}"

2. Bản chất Kỹ thuật (Bộ máy xử lý dưới Backend)
Hệ thống tuyệt đối không đọc file Word như một trang giấy.
Sự thật kỹ thuật có thể kiểm chứng: Bản chất file .docx chỉ là một file nén .zip. Backend sẽ dùng code Python âm thầm giải nén file này ra, chui vào đường dẫn lõi word/document.xml để đọc mã nguồn XML của tài liệu.

Tại đây, module word_scanner.py sẽ dùng Biểu thức chính quy (Regex) quét toàn bộ file:

Nó dùng Regex \{\{\s*([a-zA-Z0-9_]+)\s*\}\} để "vợt" tất cả các biến đơn (như tong_so_vu_chay). Thấy chữ tong_ hoặc so_, nó tự động gán kiểu dữ liệu là number.

Nó dùng Regex quét các thẻ lặp {% for ... in ([a-zA-Z0-9_]+) %} để vợt các mảng (như danh_sach_cnch). Nó tự động gán kiểu dữ liệu là array.

3. Dữ liệu Đầu ra (Lưu chết vào Database)
Sau khi quét xong, hệ thống không lưu file Word vào CSDL. File Word được băm tên ném lên kho MinIO/S3. Thứ được lưu vào PostgreSQL (Bảng extraction_templates) là 2 cục JSONB định hình luật chơi cho toàn bộ hệ thống sau này:

CỤC 1: Cột schema_definition (Luật cho AI bóc tách)
Cục này sinh ra để làm vòng kim cô ép con AI (Gemini) ở Bước 2 phải tuân thủ.

JSON
{
  "fields": [
    {
      "name": "tong_so_vu_chay",
      "type": "number",
      "description": "Tổng số vụ cháy tự động trích xuất"
    },
    {
      "name": "danh_sach_cnch",
      "type": "array",
      "description": "Danh sách các vụ cứu nạn cứu hộ",
      "items": {
        "type": "object",
        "fields": [
          {"name": "thoi_gian", "type": "string"},
          {"name": "dia_diem", "type": "string"},

CỤC 2: Cột aggregation_rules (Luật cho Pandas cộng dồn)
Cục này đẻ ra để làm "sách hướng dẫn" cho code Backend ở Bước 3 biết cách xào nấu dữ liệu từ báo cáo ngày lên báo cáo tuần.

JSON

GIAI ĐOẠN 2: TRẬN CHIẾN BÓC PDF VÀ CHỐT KIỂM DỊCH (AI EXTRACTION & VALIDATION LAYER).

Đây là khúc "đẫm máu" nhất của hệ thống, nơi mà thế giới phi cấu trúc (các file PDF báo cáo ngày lộn xộn) va chạm với thế giới có cấu trúc (Database). Mày ném cái file Báo cáo Ngày 02/03/2026 vào, hệ thống sẽ nhai nó qua 3 bước như sau:

1. Băm File (PDF Parsing)
Hệ thống đéo ném nguyên cái file PDF nặng trịch cho con AI đọc.

Lõi Backend dùng thư viện pdfplumber để "lột da" cái file PDF KV30 BCN 02.03.26.pdf của mày.

Toàn bộ chữ nghĩa, từ dòng "1. Tình hình cháy: Không" cho đến cái bảng 61 dòng ở trang cuối, đều được băm nát và xếp lại thành một chuỗi văn bản (Markdown) siêu dài nhưng giữ nguyên được tọa độ hàng cột của bảng biểu.

2. Ép cung LLM (LLM Extraction)
Hệ thống cầm cái chuỗi văn bản Markdown đó, kẹp chung với cục "Luật pháp" schema_definition (vừa đẻ ra ở Nhịp 1) rồi quăng vào API của con Gemini.
Nó ra lệnh: "Đọc cái mớ chữ này, bốc đúng số liệu nhét vào cái khuôn JSON này cho tao. Cấm nói nhảm!"

Do bản chất LLM là mô hình ngôn ngữ, nó hay bị "ảo giác" (hallucination) định dạng. Nên cục Raw JSON nó khạc ra sẽ trông rác rưởi như thế này:

JSON
{
  "ngay_bao_cao": "ngày 02 tháng 03 năm 2026", 
  "tong_so_vu_chay": "Không", 
  "tong_so_vu_cnch": "01 sự cố", 
  "danh_sach_cnch": [
    {
      "thoi_gian": "05 giờ 10 phút ngày 02/03/2026",
      "dia_diem": "Sông Sài Gòn, phường Thủ Dầu Một",
      "luc_luong": "01 xe chỉ huy, 01 cano cùng 06 CBCS",
      "ket_qua_xu_ly": "bàn giao thi thể và hiện trường cho Công an phường",
      "thong_tin_nan_nhan": "Công an phường đang xác minh"
    }
  ]
}
Nhìn kỹ thấy rủi ro chưa? Chỗ số vụ nó trả về chữ "Không" và "01 sự cố" thay vì số nguyên. Ngày tháng thì viết nguyên câu tiếng Việt. Nếu mày cho phép lưu cục này thẳng xuống DB, đến lúc lấy ra tính tổng (SUM), server sẽ Crash ngay lập tức vì đéo thể lấy chữ "Không" cộng với "01 sự cố".

3. Chốt Kiểm Dịch (Pydantic Validation Layer)
Đây là cái khiên bảo vệ Database của mày. Tuyệt đối cấm dùng AI ở bước này, vì AI đéo có tính tất định (deterministic). Mày phải dùng đoạn code Python thuần túy mà tao đã review ở trên để làm "Hải quan ép kiểu" (Type Coercion).

Cục Raw JSON bị ném qua chốt kiểm dịch này. Đoạn code sẽ "nắn xương" nó:

Nắn Số: Thấy tong_so_vu_chay đang là chữ "Không", Regex sẽ map với bộ từ điển và ép nó thành số 0. Thấy "01 sự cố", nó lột chữ "sự cố" vứt đi, ép thành số 1.

Nắn Ngày: Thấy "ngày 02 tháng 03 năm 2026", Regex bóc tách và ép lại đúng định dạng canonical 02/03/2026.

4. Dữ liệu Đầu ra (Lưu xuống Database)
Vượt qua được chốt kiểm dịch, mớ rác đã biến thành vàng. Cục JSON này giờ đây sạch sẽ 100%, đúng kiểu dữ liệu (số ra số, ngày ra ngày, mảng ra mảng).

Hệ thống lập tức gọi lệnh INSERT, nhét cục Clean JSON này vào cột extracted_data (kiểu JSONB) của bảng extraction_jobs trong PostgreSQL:

JSON
{
  "ngay_bao_cao": "02/03/2026", 
  "tong_so_vu_chay": 0, 
  "tong_so_vu_cnch": 1, 
  "danh_sach_cnch": [
    {
      "thoi_gian": "05 giờ 10 phút ngày 02/03/2026",
      "dia_diem": "Sông Sài Gòn, phường Thủ Dầu Một",
      "luc_luong": "01 xe chỉ huy, 01 cano cùng 06 CBCS",
      "ket_qua_xu_ly": "bàn giao thi thể và hiện trường cho Công an phường",
      "thong_tin_nan_nhan": "Công an phường đang xác minh"
    }
  ]
}
Kết luận Nhịp 2: Trải qua Nhịp 2, một file PDF tĩnh đã chính thức được "số hóa" thành một dòng dữ liệu cấu trúc chặt chẽ nằm gọn trong Database, sẵn sàng để tính toán cộng trừ nhân chia.

GIAI ĐOẠN 3: NẤU CAO VÀ ĐÚC KHUÔN (AGGREGATION & WORD EXPORT). Đây là chặng cuối cùng. Lúc này, dưới Database của mày đang nằm im lìm 7 cục "Clean JSON" của 7 ngày trong tuần. Đột nhiên sếp vỗ vai: "Xuất báo cáo tuần cho anh". Mày bấm 1 nút trên Web, dưới Backend hệ thống sẽ gầm rú và chạy qua 3 nhịp cuối cùng này để ói ra cái file Word hoàn hảo.

1. Đập Phẳng và Xào Nấu (Pandas Aggregation)
Hệ thống đéo dùng câu lệnh SQL SELECT SUM() truyền thống, vì dữ liệu đang nằm lồng ghép nhiều tầng trong cột JSONB. Thằng Backend sẽ kéo cả 7 cục JSON đó lên bộ nhớ RAM.

Tại đây, thư viện chuyên trị dữ liệu lớn là Pandas ra tay:

Nó dùng hàm pandas.json_normalize() đập phẳng mọi mảng JSON lồng nhau thành một cái bảng 2 chiều (DataFrame) y như Excel.

Bắt đầu mở cuốn "Luật pháp" aggregation_rules (đã lưu ở Giai đoạn 1) ra để tra cứu:

Thấy luật {"source": "tong_so_vu_chay", "method": "SUM"}: Nó lấy cột số vụ cháy của 7 ngày cộng lại với nhau (Ví dụ: 0 + 1 + 0 + 2 + ... = 3 vụ).

Thấy luật {"source": "danh_sach_cnch", "method": "CONCAT"}: Nó đi gom tất cả các mảng chi tiết vụ việc của 7 ngày nối đuôi nhau thành một mảng siêu dài.

Thấy luật {"source": "nguoi_ky", "method": "LAST"}: Nó chốt lấy tên ông Đội trưởng ở cái báo cáo của ngày cuối cùng trong tuần.

2. Cục Data Tổng Hợp (The Final Payload)
Sau khi Pandas xào nấu xong, nó vắt ra được một cục "Cao" (Aggregated JSON). Đây là cục Data bọc thép cuối cùng, mang sứ mệnh 1-1 với cái file Word. Cục này chứa sức mạnh của 7 ngày cộng lại:

3. Đúc Khuôn Bơm Text (Headless Word Export)
Lúc này, hệ thống sẽ mò lên kho MinIO, lôi cái file Bao_cao_tuan_KV30_mau.docx (có đục sẵn các lỗ Jinja2) kéo thẳng về RAM server (sử dụng io.BytesIO). Nó không hề lưu file này xuống ổ cứng để tránh rác hệ thống.

Thư viện docxtpl bắt đầu mở file Word ngay trên RAM, cầm cục "Cao" ở Bước 2 nhồi vào:

Thấy thẻ {{ tong_so_vu_chay }}, nó xóa thẻ đó đi và nhét số 3 vào, giữ nguyên font Times New Roman 14.

Thấy thẻ {%p if tong_cong_van == 0 %}Không.{%p else %}..., nó so dữ liệu thấy số 0, nó sẽ in ra đúng chữ "Không.", sau đó nó âm thầm xóa bỏ mọi dấu vết (khoảng trắng, dòng thừa, enter thừa) của khúc else bên dưới để văn bản không bị thủng lỗ chỗ.

Gặp cái bảng 61 dòng ở cuối trang có thẻ {% tr for row in bang_thong_ke %}, nó copy cái dòng <w:tr> (Table Row) trong mã XML của Word ra làm 61 bản sao y hệt nhau, giãn khung bảng ra cực mượt, rồi nhồi số của 61 chỉ tiêu vào.

4. Xả File Về Máy (Streaming)
Kết thúc quá trình nhồi nhét, docxtpl đóng gói file Word lại. Server Backend mở một đường ống HTTP, xả thẳng luồng Byte Stream này về máy tính của mày.

Trình duyệt của mày nảy lên cái thông báo tải về: Bao_Cao_Tuan_18_12.docx. Mày mở lên, một bản báo cáo dài 5 trang, nét căng từ dấu chấm phẩy, bảng biểu thẳng tắp không trật một milimet. Sếp chỉ việc ký.
